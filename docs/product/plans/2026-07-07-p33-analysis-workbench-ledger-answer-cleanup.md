# P33 Analysis Workbench Ledger Answer Cleanup

Date: 2026-07-07

## Why P33 Exists

P32 made Analysis Workbench capable of collecting several safe evidence tasks for one business question. Real use after P32 showed the next issue: the evidence path is stronger, but the final user-facing answer can still feel template-like or be polluted by old product-result logic. Fast facts may skip the model-written answer path, `Product Result Builder` can still behave like an answer rewriter in some cases, and old consistency helpers can read raw execution rows or internal task metadata instead of the clean evidence ledger.

P33 is the phase that finishes the Analysis Workbench answer path before external tool integrations. The goal is not to add more rules. The goal is to make the current product path simpler:

```text
understand the question
-> ground it to the current dataset
-> collect evidence with tools
-> write the final Chinese answer with an LLM from the evidence ledger
-> validate only key facts
-> assemble the product result
```

Old paths that conflict with this design should be deleted instead of preserved. Historical docs can stay.

## Product Goal

Analysis Workbench should answer like a real business analyst:

- use tools to calculate facts;
- let the model write the final explanation and recommendation;
- keep hard facts evidence-bound;
- avoid fixed answer templates;
- avoid leaking raw fields, task ids, SQL, trace paths, provider metadata, or raw rows into the main answer;
- keep Report Center independent on `ReportPlan + ReportEvidencePack + EvidenceLedger + ReportDocument`.

The final answer for `fast_fact`, `standard_analysis`, and `deep_evidence` must be generated by an LLM from the clean Analysis Workbench Evidence Ledger. The model can be asked for a short factual answer on fast facts and a fuller business answer on standard/deep questions, but the user-facing conclusion must not be stitched by deterministic templates.

## Design Principles

### 1. No Template-Driven Final Conclusions

Code must not produce user-facing business conclusions such as `最高的是 X，数值为 Y` by string templates. Deterministic code may create safe labels, display values, evidence summaries, progress states, and validation messages, but the final answer fields shown as the conclusion must come from the answer LLM.

Allowed deterministic work:

- field/metric matching;
- SQL safety review;
- evidence calculation;
- evidence ledger construction;
- fact validation;
- internal metadata cleanup;
- product payload assembly.

Not allowed:

- using raw rows to compose the final conclusion;
- route-specific answer templates;
- `Product Result Builder` rewriting the business answer;
- old answer consistency helpers inventing prose from execution rows;
- keyword-heavy business-rule trees that decide meaning from fixed phrases.

### 2. Combine Question Understanding And Business Lens Where They Overlap

Question Understanding and Business Lens currently share responsibility for intent, metric, dimension, source table, and time口径. P33 should consolidate the overlapping work into one clean front-door surface:

```text
Question + current workspace profile + semantic layer
-> structured analysis intent
-> dataset-grounded metrics, dimensions, source fields, time policy, route inputs
```

The implementation can keep separate files if that is cleaner, but the product path should behave as one logical stage: understand the user's natural language and bind it to the current dataset. The model handles varied wording; deterministic code validates against the semantic layer and data profile.

### 3. Route By Evidence Complexity, Not Keywords

Standard vs Deep must not be decided by fixed words such as `建议`, `风险`, `原因`, or `优先级`. Route by the amount and type of evidence needed.

`Standard Analysis`:

- one main subject;
- one primary source table;
- one primary metric/dimension question;
- one core evidence task is enough, or multiple same-table fields can be answered by one safe query;
- no cross-evidence tradeoff is required.

Examples:

- `最近30天哪个渠道收入最高？`
- `本月哪个门店销售额最高？`
- `各品类销售额排名怎么样？`

`Deep Evidence`:

- two or more core evidence tasks are required;
- or two or more source tables are required;
- or several metrics must be compared as a tradeoff;
- or the question asks for a decision, reason, risk, recommendation, or priority that cannot be responsibly answered from one fact table.

Examples:

- `最近30天哪个渠道最值得加预算？`
- `结合收入、投放花费和客服投诉，哪个渠道最需要优化？`
- `哪些门店销售高但售后问题也高？`

Suggested conservative routing rule:

```text
if evidence_task_count >= 2:
  route = deep_evidence
elif source_table_count >= 2:
  route = deep_evidence
elif supporting_metric_count >= 2 and the answer needs a tradeoff/recommendation:
  route = deep_evidence
else:
  route = standard_analysis
```

The counts should come from the structured analysis intent and evidence task plan, not from keyword matching.

### 4. Evidence Ledger Is The Only Answer Input

The answer LLM should receive a clean Analysis Workbench Evidence Ledger, not raw SQL rows or task-run internals. The ledger should contain business-readable facts, derived metrics, data limits, table/evidence refs, time口径, and chart refs.

The ledger should exclude or hide from the answer prompt:

- SQL text;
- raw execution rows unless transformed into safe facts;
- `task_id`, `task_purpose`, ledger ids, trace ids, provider metadata;
- local paths, API keys, database paths;
- raw ECharts option or chart spec JSON.

If the model needs row-level context, it should receive a sanitized evidence table with business labels and units, not internal fields.

### 5. Product Result Builder Becomes An Assembler

`Product Result Builder` should assemble the product response. It should not generate, rewrite, or template the business answer.

It may:

- pass through `business_answer`;
- attach `question_evidence_ledger` summaries;
- attach chart artifacts;
- attach progress and technical details;
- hide internal details from the main UI;
- rebuild historical records only through the same current assembler contract.

It must not:

- compose final answer prose from execution rows;
- override LLM-written conclusions with deterministic wording;
- use old consistency helpers as a second answer generator;
- expose internal ids or raw task metadata in the main answer.

Old helper code that only exists to rewrite business conclusions from raw rows should be removed or downgraded to validation/scrubbing. If a helper remains useful, it should operate on the ledger and only clean unsupported facts or leaks.

### 6. Chart Generation Can Run In Parallel

After the evidence ledger is ready, chart generation can run alongside answer generation when useful:

```text
Evidence Ledger
-> Business Answer LLM
-> Fact Checker / minimal repair

Evidence Ledger
-> Visualization Agent / ChartArtifact / ECharts
```

Fast facts should only generate charts when explicitly requested. Standard/deep answers may generate charts when the evidence is chartable and useful. The chart generator should consume the same clean evidence ledger or sanitized chart table, not raw task rows.

## Target Analysis Workbench Path

```text
user question or same-thread follow-up
-> Question Understanding + Business Lens combined grounding
-> clarification if required
-> route by evidence complexity:
   - fast_fact: one compact evidence task, LLM short answer
   - standard_analysis: one main evidence path, LLM business answer
   - deep_evidence: multiple evidence tasks, LLM business answer
-> SQL/evidence tools:
   - one reviewed SQL per task
   - SQL review before execution
   - evidence validation
-> Analysis Evidence Ledger
-> Business Answer LLM from ledger only
-> Fact Checker for key hard facts only
-> Minimal Repair for unsupported facts or internal leaks only
-> optional parallel ChartArtifact/ECharts generation from ledger
-> Product Result Builder assembler
-> Analysis Thread Memory/history/front-end display
```

## What To Delete Or Retire

P33 may delete old paths instead of preserving compatibility when they conflict with the current product direction:

- final-answer templates for fast facts or standard answers;
- raw-row answer composers;
- `Product Result Builder` branches that rewrite conclusions;
- consistency helpers that act as hidden answer generators;
- stale tests that assert template wording;
- old route paths that bypass the evidence ledger;
- compatibility code for deleted clarification/follow-up flows;
- internal-id display paths in main UI payloads.

Keep historical docs and tests that explicitly protect deleted-path absence.

## Acceptance Criteria

P33 is complete when:

- every Analysis Workbench final conclusion is model-written from the clean evidence ledger; deterministic repair may scrub unsupported details/internal leaks or normalize provider structure, but it must not generate a replacement business conclusion from ledger facts, raw rows, or templates;
- fast facts are short and fast but not template-composed;
- Standard vs Deep routing is based on evidence complexity, not keyword templates;
- Evidence Ledger is the only input source for user-facing answer generation;
- Product Result Builder is an assembler, not a hidden answer generator;
- old conflicting paths and stale template tests are deleted or rewritten as negative boundary tests;
- chart generation does not pollute or delay the answer path unnecessarily;
- Report Center remains independent;
- real DeepSeek tests show question, route, elapsed time, answer, evidence, chart behavior, leak checks, and Report Center independence.
